## 官方文档里的“暗坑”：为什么你的钱会白花？

我们先不说平台,先聊聊官方那份几乎“原罪级”的计费文档。如果你直接按文档示例去调 API,大概率会碰到以下几个问题：

**1. 模型ID的“障眼法”**

官方文档中,示例代码里直接写死了模型 ID,比如 `gpt-5-turbo-beta`。但它没有说清楚,这个 ID 可能在某个时间点后失效,取而代之的是一个随机生成的、只有官方才懂的 UUID。

**后果**：你的代码里写死了 ID,一旦失效,API 直接返回 404。你以为自己代码写错了,反复调试,白白浪费大量的时间和请求次数。而请求次数,可是要收费的。

**2. Token 计费的“暗箱操作”**

官方文档里的计费公式,表面上写的是“总 token 数 × 单价”。但它没告诉你,这里的“总 token 数”可能包括了系统提示词、特殊分隔符,甚至是你请求头里的一些隐蔽信息。

**后果**：你明明只发了 100 个 token 的提示词,最终被计费的可能是 200 个 token。你完全没有感知,但账户余额却在飞速下降。这就是典型的“隐形成本”。

**3. 接口验签的“高门槛”**

官方文档要求开发者实现一套极其复杂的 HMAC-SHA256 签名验签流程。文档里充斥着晦涩的术语和未公开的算法细节,错误率极高。很多新手在这第一步就被劝退了,或者在调试过程中反复返回 `401 Unauthorized` 错误,每次错误都在消耗你的免费额度。

这些“坑”不是偶然的。它们的设计初衷,就是让普通开发者难以“白嫖”到稳定的服务,从而逼你去购买更昂贵的套餐或企业版。
