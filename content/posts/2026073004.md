## 改造后：异步模型为什么“腰斩”了成本

我接下来说的改造，不需要你换语言，不需要你换框架。只需要把 Java 的 `HttpClient` 从 `send()` 换成 `sendAsync()`，再用 `CompletableFuture` 做一下编排。

改造后的核心逻辑大概是这样：

java
// 使用 Java 11 内置的 HttpClient，直接设置为异步模式
HttpClient client = HttpClient.newBuilder().build();

public CompletableFuture<String> translateAsync(String text) {
    // 这里构建请求，然后把 send 改成 sendAsync
    HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create("https://www.qianjuai.com/v1/messages")) // [千聚ai聚合站](https://www.qianjuai.com/)的 Claude 接口
            .header("Authorization", "Bearer " + apiKey)
            .POST(...)
            .build();
    // sendAsync 立刻返回，不会阻塞当前线程
    return client.sendAsync(request, BodyHandlers.ofString())
                 .thenApply(response -> extractContent(response.body()));
}


`sendAsync()` 返回 `CompletableFuture`。这个 future 会在网络响应真正到达时 “完成”，并主动通知你的后续处理逻辑。

**核心优势来了：这个线程不会在等待网络 I/O 时白白挂起了。** 在[千聚ai聚合站](https://www.qianjuai.com/)的 Claude Haiku 处理你的请求、返回结果的那 2 到 3 秒里，你的 CPU 完全可以去处理其他请求，或者做自己的业务逻辑。

用上[千聚ai聚合站](https://www.qianjuai.com/)的 API，配合异步模型，逻辑就变成了——

- 一个线程发出异步请求。
- 等响应时，这个线程立刻回去接下一个任务。
- 响应回来后，自动执行后续结果处理。

换句话说：**你不再需要用 CPU 的时间去等 I/O。**
